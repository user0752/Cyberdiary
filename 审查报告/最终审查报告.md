## 一、架构层面问题

### A1 【高】API 层 / 外部模块调用私有函数 — 分层破坏

**状态**：✅ 确认存在

- `backend/app/api/v1/multi_agent_compile.py` 第 400 行：`compile_service._parse_compile_output(...)` — API 层直接调用 service 模块的下划线私有函数
- `backend/app/services/multi_agent_graph.py` 第 21 行：`from app.services.compile_service import _safe_progress_update` — 跨模块导入私有函数

**风险**：私有函数（`_` 前缀）不受 API 稳定性承诺保护，内部重构会导致连锁破坏。已形成隐式强耦合。

**修复**：将 `_parse_compile_output` 重命名为 `parse_compile_output`（公共接口），`_safe_progress_update` 提取到独立的 `progress_tracker.py` 模块。

### A3 【高】默认关闭鉴权 — 公网部署裸奔

**状态**：✅ 确认存在

- `backend/app/core/config.py` 第 46 行：`auth_mode: str = "none"`

虽然项目支持 JWT（`deps.py` 有完整 JWT 验证逻辑），但默认关闭。直接部署公网时所有 API 无鉴权保护。

**修复**：生产环境默认 `auth_mode=jwt`；Docker 部署文档强制要求设置认证；启动时若检测到公网暴露且无鉴权则告警。

## 二、代码层面问题

### C1 【高·数据丢失】HumanReviewTask 模型字段与代码不匹配

**状态**：✅ 确认存在 — 但严重程度需修正

> ⚠️ **前期报告误判**：`review_report2.md` 声称这是"运行时必崩 `AttributeError`"，但经验证，SQLAlchemy ORM 实例允许设置非映射列的 Python 属性（通过 `object.__setattr__`），**不会抛出 `AttributeError`**。

**实际问题**：

`human_review_manager.py` 第 87-90 行设置的字段与模型定义不匹配：

| 代码中使用 | 模型中实际字段 |
|-----------|-------------|
| `t.decision` (L87, L122) | 不存在 |
| `t.edited_wiki` (L89, L124) | 不存在 |
| `t.resolved_at` (L90, L125) | 不存在 |

模型 `multi_agent.py` 定义的实际字段：`user_edited_content`、`revised_content`、`final_content`、`decided_at`。

**实际后果**：`decision`、`edited_wiki`、`resolved_at` 被设置为普通 Python 属性 → 数据库 `commit()` 时**这些字段的值不会被持久化** → 审核决策数据静默丢失。

**修复**：统一字段命名。推荐方案：
- 在模型 `HumanReviewTask` 中添加 `decision`、`edited_wiki`、`resolved_at` 字段
- 或修改 `human_review_manager.py` 使用 `user_edited_content` / `decided_at`

### C2 【高】Agent 错误被静默吞噬 — Fallback 伪装成功

**状态**：✅ 确认存在

`agent_error_handler.py` 第 51-61 行 `_fallback()` 方法：

```python
if "reviewer" in agent:
    state.setdefault("reviews", [])
    state["reviews"].append({"score": 8.0, "feedback": "auto-pass (fallback)"})
```

当 reviewer agent 因熔断/超时/异常失败时，fallback 给出 8.0 分（正好是通过阈值），编译引擎显示"成功"但 Wiki 质量未经评审。

**修复**：在编译结果中标记使用了 fallback 的 agent；在 fallback 分数超过阈值时标记为"降级质量"。

### C3 【高】LLMCache 全局锁串行化 + 连接未关闭

**状态**：✅ 确认存在

- `llm_cache.py` 第 21 行：`self._lock = asyncio.Lock()` — 所有 `get/set` 操作串行化执行
- 整个类没有 `close()` 方法 — aiosqlite 连接在进程退出时不会显式关闭
- 第 53 行 `json.dumps(kw, sort_keys=True)` — 若 kwargs 包含不可序列化对象会抛 `TypeError`

**修复**：使用连接池或 per-connection SQLite WAL 模式；添加 `async def close()` 并在 lifespan shutdown 中调用；缓存 key 生成包裹 `try/except`。

### C4 【高】Chat 流式响应历史消息处理有隐患

**状态**：✅ 确认存在 — 但代码已有部分改进

`chat_service.py` 第 197 行：
```python
messages = [...] + history_msgs[:-1] + [{"role": "user", "content": message}]
```

`[:-1]` 会丢弃最后一条历史消息。代码注释说明这是为了避免重复（因为用户消息在独立 session 中已保存），但：
- 如果最后一条消息是 assistant 回复，它也会被丢弃 → LLM 上下文缺失
- `get_messages` 读取使用的是请求 session（第 178 行），而用户消息在独立 session 中已提交 → 存在读到旧快照的可能

**正面评价**：代码已在第 141 行使用独立 session 保存用户消息，这解决了原报告中提到的"SSE 断开导致数据丢失"问题。且 `build_chat_context` 函数（第 78-124 行）已有更完善的实现。

**修复**：统一使用 `build_chat_context()` 替换内联逻辑。

### C5 【中】多智能体语义链接 N×M 重复插入

**状态**：✅ 确认存在

`multi_agent_compile.py` 第 416-425 行：
```python
for link in final_state.get("suggested_links", []):
    if link.get("confidence", 0) >= 0.7:
        for page_data in pages:      # 对每个 page 执行
```

### C6 【中】标签/JSON 字段使用 LIKE 匹配 — 无索引

**状态**：✅ 确认存在

- `memo_service.py` 第 61 行：`Memo.tags.like(f'%"{tag}"%')`

- `compile_service.py` 第 406 行：`WikiPage.tags.like(f'%"{tag}"%')`

### C7 【中】link_pairs 数量 O(N²) 爆炸 + OR 子句爆炸

**状态**：✅ 确认存在

`compile_service.py` 第 391-397 行：对每对同 tag 的 slug 生成双向链接，复杂度 O(N²)。后续第 420-424 行用 `or_(*[...])` 构建动态查询检查已存在链接，若 link_pairs 数量大则 SQL 包含数百个 OR 子句。

**修复**：限制同 tag 最大关联数；分批插入；使用 IN 子句替代 OR 链。

### C8 【中】CoordinatorAgent KMeans 缺少并发控制

**状态**：⚠️ 部分确认 — 不会崩溃但有性能风险

`coordinator_agent.py`：
- 第 15-18 行：`if embedding_model and len(memo_ids) >= 3` 守卫条件确保只在 memo ≥ 3 时才调用聚类
- 第 18 行：`n_clusters=min(3, len(memo_ids) // 2 + 1)` — n_clusters 被限制 ≤3，且 `len(memo_ids) >= 3` 确保不会出现 `n_clusters > len(memo_ids)`
- **但**：`_batch_embed` 使用 `asyncio.gather` 批量调用 embedding，无 `asyncio.Semaphore` 限流 → memo 多时可能触发 provider rate limit

**修复**：添加 `asyncio.Semaphore` 限制并发数；聚类失败时 fallback 到单组。

## 三、安全层面问题

### S1 【高】API Key 加密密钥与 JWT 密钥同源

**状态**：✅ 确认存在

`security.py`：
- JWT 签名（第 24 行）：`jwt.encode(to_encode, settings.secret_key, ...)`
- AES 密钥派生（第 43 行）：`hashlib.pbkdf2_hmac("sha256", settings.secret_key.encode(), ...)`

虽然 AES 通过 PBKDF2 派生而非直接使用原始 `SECRET_KEY`，但派生源相同 → 若 `SECRET_KEY` 泄露，攻击者可重现派生过程解密所有存储的 API Key。

**修复**：使用独立的 `ENCRYPTION_KEY` 环境变量。

## 八、整体结论

**综合评级：有条件通过**

项目在功能层面已经成型，架构分层（API → Service → Model）有基本雏形，代码风格较一致，且有意识地引入了熔断、重试、缓存、FTS5、Alembic 等生产化概念。

**核心风险（P0，上线前必须解决）**：
1. **HumanReviewTask 字段不匹配**（C1）→ 审核决策数据静默丢失
2. **默认关闭鉴权**（A3）→ 公网部署等于裸奔
3. **Docker/Nginx 开发配置**（A4）→ 无法实际部署
4. **全局内存状态**（A2）→ 多实例不可行
5. **核心模块无测试**（M1）→ 无法保证重构不引入回归

**可行的上线路线**：
- P0 修复：约 1-1.5 周
- P1 加固：约 1-2 周
- P2 优化：持续进行

当前项目**适合作为个人本地工具或技术演示**，完成 P0+P1 后可达到内网部署/小范围测试水平，完成全部改进后可考虑公网部署。
