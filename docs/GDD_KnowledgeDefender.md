# Knowledge Defender — 知识防御战 GDD

> 版本：v1.0 | 日期：2026-06-21 | 状态：设计草案
> 
> 基于赛博笔记 Wiki 编译知识库的游戏化学习模块

---

## 一、核心判断：这个想法非常好

### 1.1 为什么值得做

你有三个独特优势是大多数"做个答题游戏"的人不具备的：

| 优势 | 说明 |
|------|------|
| **知识源是结构化的** | Wiki 有 front matter（标题/类型/标签/摘要），不是散落的文本 |
| **图谱关系可复用** | `wiki_links` 表已经存储了双向链接，这就是天然的游戏地图数据 |
| **LLM Pipeline 已通** | LiteLLM 路由 + SSE 流式 + Prompt 模板体系，出题成本几乎为零 |

### 1.2 认知科学支撑

- **主动回忆（Active Recall）** — 比被动阅读的记忆保留率高 2-3 倍（Roediger & Karpicke, 2006）
- **测试效应（Testing Effect）** — 答题本身就是最强记忆方式，不论对错
- **间隔重复（Spaced Repetition）** — 结合艾宾浩斯曲线，遗忘→重考→加固
- **可变奖励（Variable Reward）** — 塔防的波次随机性触发多巴胺循环

### 1.3 现有 MVP 的差距分析

你已经完成的 MVP（GameArena.vue + game_service.py）是**选择题问答器**，它的核心问题：

| 现有状态 | 缺少什么 |
|---------|---------|
| 选择题单项问答 | 没有时间压力、没有失败惩罚 |
| 静态题目列表 | 没有动态难度、没有关卡感 |
| 正确/错误反馈 | 没有"再来一次"的驱动力 |
| 结果页展示 | 没有进度可视化、没有成就系统 |

**游戏化不是加个皮，而是改变学习的心流状态。**

## 二、Design Pillars（设计支柱）

任何设计决策都对照这三条：

### Pillar 1 — 知识即武器（Knowledge as Ammo）
玩家不是在"答题"，而是在"开火"。答对 = 防御塔开炮。知识掌握度越高，炮塔越强。

### Pillar 2 — 遗忘即敌人（Forgetting as Threat）
敌人不是随机生成的，而是来自你 Wiki 知识图谱中**你正在遗忘的概念**。遗忘曲线驱动敌人生成频率。

### Pillar 3 — 一分耕耘一分收获（Effort → Visible Progress）
每局游戏结束后，玩家必须看到可视化的进步——知识掌握度提升、盲区缩小、新概念解锁。不成就不结束。

## 三、Gameplay Loop 设计

### 3.1 Core Loop

```
┌──────────────────────────────────────────────────┐
│                 MOMENT-TO-MOMENT (0-30s)          │
│                                                   │
│  敌人靠近  ──→  点击防御塔  ──→  弹出题目  ──→    │
│  答对：塔开火，敌人扣血/死亡                      │
│  答错：敌人前进一格，Socratic 提示出现             │
│                                                   │
└──────────────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────┐
│                 SESSION LOOP (5-15min)            │
│                                                   │
│  波次1-3：复习最近学过的概念（Easy）               │
│  波次4-6：混合已掌握 + 薄弱概念（Medium）          │
│  波次7-9：攻击知识盲区（Hard）                    │
│  波次10：BOSS 战（综合应用题）                     │
│                                                   │
│  胜利条件：10波全部清除                           │
│  失败条件：生命值归零（20条命，敌到达终点扣命）   │
│                                                   │
└──────────────────────────────────────────────────┘
                         │
                         ▼
┌──────────────────────────────────────────────────┐
│                 LONG-TERM LOOP (days-weeks)       │
│                                                   │
│  每日挑战：基于遗忘曲线推荐今日地图               │
│  知识域解锁：掌握度 > 70% 解锁下一知识域地图      │
│  成就系统：连续7天挑战、全概念五星、BOSS无伤       │
│  排行榜（可选）：好友间正确率/速度对比             │
│                                                   │
└──────────────────────────────────────────────────┘
```

### 3.2 敌人类型设计

敌人都来自你的 Wiki 知识图谱节点，不同类型的 Wiki 节点映射不同敌人行为：

| 敌人类型 | 对应 Wiki 类型 | 行为 | HP | 速度 | 答题时限 |
|---------|---------------|------|-----|------|---------|
| **Forgetter（遗忘者）** | concept（概念） | 直线前进，基础题型 | 低 | 慢 | 30s |
| **Confuser（混淆者）** | comparison（对比） | 出"区分A和B"题，有干扰项 | 中 | 中 | 20s |
| **Blinder（盲区怪）** | entity（实体） | 玩家历史错题重出 | 中高 | 快 | 15s |
| **BOSS（知识领主）** | synthesis（综合） | 多知识点综合应用题 | 极高 | 极慢 | 45s |

### 3.3 防御塔类型设计

每座塔对应一个 Wiki 概念。塔的属性由玩家对该概念的掌握度决定：

| 塔属性 | 计算方式 | 范围 |
|--------|---------|------|
| 攻击力 | `mastery_score × 100` | 0-100 |
| 射程 | `80 + mastery_score × 40` | 80-120px |
| 攻速 | `0.5 + mastery_score × 1.0` | 0.5-1.5s/发 |
| 升级条件 | 掌握度达到阈值（0.3/0.6/0.9） | 3级 |

## 四、技术架构设计

### 4.1 渐进式方案（强烈推荐）

不要一步到位做 Phaser 塔防。分三个阶段，每个阶段都可交付、可展示：

```
Phase A: 竞速答题模式   (1-2周)   ← 在现有 MVP 上改
Phase B: 塔防核心        (3-4周)   ← 引入 Phaser 或纯 Canvas
Phase C: 地图自动生成    (1-2周)   ← Wiki 图谱 → 游戏地图
```

### 4.2 Phase A 详细设计：竞速答题模式

在现有 `GameArena.vue` 基础上，添加三个游戏化要素：

**倒计时器**：每道题 15-30 秒倒计时，超时 = 答错
**生命值系统**：3 条命，答错扣命，0 命游戏结束（现有 lives 字段已有，未使用）
**连击奖励**：连续答对 3 题 → 额外加分，连续 5 题 → 双倍分

```typescript
// 只需修改 GameArena.vue 和 game_store.ts，不需要新后端接口
interface GameState {
  timerSeconds: number       // 倒计时
  lives: number              // 生命值（已有字段）
  combo: number              // 连击数
  comboMultiplier: number    // 连击倍率
  score: number              // 分数（已有字段）
}
```

**这个阶段的价值**：验证"时间压力 + 失败惩罚"是否比纯问答更能驱动学习行为。数据说话。

### 4.3 Phase B 详细设计：塔防核心

引入 Canvas/Phaser 层，不做完整塔防，只做核心塔防机制：

**简化地图设计**（单人单路，类似 Plants vs Zombies 单行）：

```
[入口] ──→  ● ──→  ● ──→  ● ──→  ● ──→  ● ──→ [基地/生命值]
            塔1   塔2   塔3   塔4   塔5
```

- 5 个塔位，对应 5 个 Wiki 概念（从当前知识域选择）
- 点击塔 → 弹出题目 → 答对塔开火
- 敌人沿路径移动，到达基地扣命
- 每波 3-5 个敌人，每局 5-10 波

**技术选型建议**：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **纯 Canvas 2D** | 零依赖，完全可控，和 Vue3 集成简单 | 需要自己写渲染循环 | ⭐⭐⭐⭐⭐ |
| Phaser 3 | 成熟框架，物理/动画/场景管理完善 | 重（~1MB），Vue3 集成有坑 | ⭐⭐⭐ |
| PixiJS | 轻量 WebGL 渲染 | 只有渲染，没有游戏逻辑 | ⭐⭐⭐ |
| CSS animation + DOM | 最简单，和 Vue 天然一体 | 表现力有限 | ⭐⭐⭐ |

**推荐用纯 Canvas 2D**。原因：
1. 你的塔防需求很简单（单路、5塔位、波次出怪），不需要 Phaser 级别的物理引擎
2. Canvas 和 Vue 3 的组合已经在 `ForceGraph.vue`（知识图谱）里验证过了
3. 学习成本低，维护成本低，打包体积小（适合 PWA）

### 4.4 Phase C：Wiki 图谱 → 游戏地图自动生成

这是最酷的部分。你的 `wiki_links` 表已经存储了概念间的关系：

```
Wiki 关系图谱              游戏地图
─────────────────────    ─────────────────
概念A → 概念B            依赖关系 → 敌人的行进路线
概念的"依赖链"            前驱概念不掌握 → 敌人更强
图谱深度                   地图难度
```

**自动生成算法**：
```python
def wiki_graph_to_game_map(wiki_domain: str) -> GameMap:
    """
    1. 从 wiki_links 表中取出该领域的子图
    2. 按依赖关系拓扑排序 → 敌人顺序
    3. 深度 > 3 的概念链 → 多波次地图
    4. 玩家掌握度低的概念 → 放在关键塔位
    5. 生成 JSON 地图配置，前端 Canvas 渲染
    """
```

## 六、具体设计建议

### 6.1 不要做的事（基于游戏设计经验）

| 不要 | 原因 |
|------|------|
| ❌ 一步到位做完整塔防 | 核心乐趣（答题→杀敌）没验证之前，投入 Phaser 是浪费 |
| ❌ 过度设计 Agent 系统 | 你的塔防手册里设计了 5 个 LangGraph Agent，但 MVP 阶段一个 QuestionAgent 就够 |
| ❌ 追求"职业级"视觉 | 学习工具的视觉目标是"清晰"不是"炫酷"，CSS 极简风已经很好 |
| ❌ 引入 Socket.IO | 除非要做多人对战，单人塔防不需要 WebSocket |

### 6.2 应该做的事

| 要做 | 理由 |
|------|------|
| ✅ 先加倒计时和生命值 | 改动量极小（前端加个 timer，复用现有 lives 字段），但游戏感立刻出来 |
| ✅ 加音效反馈 | 答对"叮"、答错"嗡"、连击音阶上升，听觉反馈 = 多巴胺触发器 |
| ✅ 记录 response_time | 已经记录答题对错，加个反应时间，可以做速度分析（快且准 vs 慢但准） |
| ✅ 加每日挑战 | 基于 knowledge_mastery 表（手册已设计），自动推荐今天该复习的概念 |

## 八、总结

你的想法方向完全正确，而且你已经有了 80% 的基础设施。现在的关键不是"做什么"，而是"先做什么"。

**核心建议：先做 Phase A（竞速答题），用一周验证"时间压力 + 失败惩罚"是否让学习更有趣。** 如果数据好（日均答题数提升、正确率不降反升），再投入 Phase B 做 Canvas 塔防。

记住：游戏化的本质不是界面更好看，而是让**学习的反馈循环**变得跟游戏一样紧凑——每 30 秒一次小反馈，每 15 分钟一次大反馈，每天一个挑战目标。
