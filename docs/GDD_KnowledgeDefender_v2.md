## 一、GitHub 上值得参考的现代开源网页游戏

这次搜索排除了老项目（2022 年前的）、小项目（star < 100）、以及 Unity/Godot 等非网页项目。以下是筛选结果：

### 方案 A：知识卡牌对战（Card Roguelike Battle）⭐ 强烈推荐

**参考项目：Slay the Web**

| 元信息 | 值 |
|--------|-----|
| 仓库 | [oskarrough/slaytheweb](https://github.com/oskarrough/slaytheweb) |
| 技术栈 | Astro + TypeScript + Preact + HTM + Vite + Bun |
| 授权 | MIT |
| 最后更新 | 2026 年 5 月（持续活跃） |
| 在线 Demo | slaytheweb.cards（可直接玩） |

**为什么推荐：**
- 它是《Slay the Spire》的网页版完整复刻——卡牌构筑、敌人、地图、遗物全都有
- 架构极其清晰：`src/game/` 纯游戏逻辑（脱离 UI 可独立测试）+ `src/ui/` 展示层
- 技术栈轻量——**没有 React，没有 Vue**，就 Preact + HTM + Vite
- 核心战斗循环拿来就能改：把"出牌"改为"出题"，把"消耗能量"改为"答题" + 计时器
- MIT 协议，可随意 fork 修改

**改造成知识游戏的思路：**

```
Slay the Web 原有          → 你的改造
──────────────────────    ────────────────────
抽牌 → 点牌 → 打出去       → 抽卡 → 弹出题目 → 答题正确则放技能
卡牌效果（攻击/防御）       → 题目类型决定效果（选择题=攻击，判断题=防御）
敌人意图（进攻/防守）       → 敌人由薄弱知识概念驱动
遗物效果                   → 连续答题的 buff/debuff
地图节点（怪物/宝箱/篝火）  → 知识域节点（新概念/复习/挑战Boss）
```

**改动量评估**：中等（2-3周）
- 保留核心战斗循环、UI、地图、动画
- 替换卡牌效果引擎 → 对接你的 game_service.py 出题 API
- 替换敌人 AI → 读取 knowledge_mastery 数据决定敌人类型
- 数据层对接：Wiki 概念 (wiki_page_id) → 卡牌，wiki_links → 地图依赖关系

### 方案 B：文字冒险探索（Text Adventure / Interactive Fiction）

**参考项目：A Dark Room**

| 元信息 | 值 |
|--------|-----|
| 仓库 | [doublespeakgames/adarkroom](https://github.com/doublespeakgames/adarkroom) |
| 技术栈 | 纯 JavaScript + HTML + CSS（零框架） |
| 授权 | **MPL 2.0**（Mozilla 公共许可，允许商用） |
| Star | **~7,000** |
| 最后更新 | 2024 年 |
| 热门度 | 无数媒体报道，移植到 iOS/Android |
| 在线 Demo | adarkroom.doublespeakgames.com |

**为什么推荐：**
- **零依赖，纯 JS/HTML/CSS**——和你现有的 Vue 3 + Vite 栈兼容完美
- 代码量极小（~2,000 行），整个游戏就在几个文件里，**一个人一周就能读完整个代码**
- 游戏机制极简：文字描述 + 按钮选择 + 资源管理 —— 非常适合改成"答题推进"模式
- MPL 2.0 协议约束少，改完放自己项目里没问题
- **你的 wiki 链接本身就是"故事地图"**：每个 Wiki 概念就是一个"房间"，wiki_links 就是房间之间的门

**改造成知识游戏的思路：**

```
A Dark Room 原有           → 你的改造
──────────────────────    ────────────────────
生火取暖 → 探索新区域       → 选择知识域 → 解锁该域下的概念
建筑（小屋/工坊/城堡）      → 概念难度等级（概念/实体/对比/综合）
外来者（陌生人/商人）       → Boss 关卡入口
资源管理（木材/铁/食物）    → 知识掌握度 + 答题消耗"注意力"资源
每一次决策都有后果          → 每道题选错 -> 损失进度/生命值
```

**改动量评估**：低（1-2周）
- 保留叙事框架、引擎、资源管理系统
- 替换游戏事件 → 对接你所选 Wiki 域的概念
- 修改资源体系 → 注意力值（代替木材/铁/食物）
- 数据层对接：Wiki 概念 (title, content) → 房间节点

### 方案 C：知识合并解谜（Knowledge Merge Puzzle）

**参考项目：2048**

| 元信息 | 值 |
|--------|-----|
| 仓库 | [gabrielecirulli/2048](https://github.com/gabrielecirulli/2048) |
| 技术栈 | 纯 JavaScript + CSS（零框架） |
| 授权 | MIT |
| Star | **~12,500** |
| 最后更新 | 2024 年（稳定版） |

**为什么推荐：**
- **star 最高的网页游戏之一**，代码极简（~1,000 行）
- 合并机制天然适配知识层级：`概念A + 概念A = 概念B` 就是 Wiki 的依赖链
- 在 2048 框架上改动非常容易——替换数字为知识点，替换"合并"为"同层级概念合成进阶概念"
- 你的 `wiki_links` 表就是天然的合并规则库

**改造成知识游戏的思路：**

```
2048 原有                 → 你的改造
──────────────────────    ────────────────────
数字方块                  → 知识卡片（显示概念名称）
合并两个相同数字 = 大一倍  → 合并两个相同层级的概念 = 升级到上层概念
分数                     → 知识掌握度分数
4×4 网格                 → 知识领域（每个格子是一个知识点位）
游戏结束（无法继续合并）   → 无法合成更高层概念时结束
```

**改动量评估**：低（3-5天）
- 保留 2048 渲染、动画、滑动交互
- 替换 grid 数据结构 → 存储概念 ID / 掌握度
- 替换 merge 逻辑 → 调用 wiki_links 查依赖关系
- 替换 UI 展示 → 显示概念名称 + 掌握度进度条

### 方案 E：知识翻牌配对（Match Memory）

**参考项目：Write your own**

（这类游戏太简单了，不需要外部项目。一个 Memory Card 游戏，纯 Vue 3 + CSS 大概 200 行就可以写完。）

**改造成知识游戏的思路：**
- 卡片背面：概念名称
- 配对规则：两个 **有链接关系的 Wiki 概念**（来自 `wiki_links` 表）
- 翻开两张卡，如果它们是对应关系（A → B），配对成功
- 配对成功 → 增加该概念的掌握度
- 配对所有概念 → 过关

**改动量评估**：极低（3天）
- 纯前端组件，不需要改后端
- 直接从 `wiki_links` 表读取配对数据

## 三、为什么我最终推荐 Card Battle（知识卡牌对战）

之前推荐塔防是太局限了。回过头来看，卡牌构筑游戏（deckbuilder roguelike）和你的项目简直是天作之合：

| 你的项目特性 | 卡牌游戏的对应 |
|------------|--------------|
| Wiki 有 5 种类型（concept/entity/comparison/synthesis/source） | 五种卡牌职业/派系 |
| wiki_links 表存双向关系 | 卡牌之间的组合/连击系统 |
| knowledge_mastery（掌握度） | 卡牌升级/强化系统 |
| compile_engine（LLM 编译） | 遇到新概念 → 获得新卡 |
| 遗忘曲线 | 每局随机分配薄弱知识作为敌人 |
| chat 对话上下文 | 答错后 AI 导师即时解释 |

### Slay the Web 的具体改造路线

```
Day 1-2: 环境搭建
  - fork 仓库，pnpm install，本地跑通
  - 读通 src/game/（战斗引擎）+ src/content/（卡牌/敌人数据）
  - 替换卡牌数据为 Wiki 概念名（先写死 10 张）

Day 3-5: 答题替换出牌
  - 修改 src/game/cards/：出牌时改为调用 LLM 出题 API
  - 答对 = 执行卡牌效果（attack/block），答错 = 卡牌失效
  - 对接你的 game_service.py 的出题端点

Day 6-10: 敌人与地图改造
  - 敌人改为来自 knowledge_mastery 的薄弱概念
  - 地图节点改为 Wiki 域：新概念（获得新卡）/ 复习（强化旧卡）/ Boss（综合题）
  - 波次难度从 wiki 图深度计算

Day 11-15: 数据持久化
  - 写 game_result 回 knowledge_mastery 表
  - 游戏结束后更新概念掌握度
  - 根据遗忘曲线决定每日推荐地图

Day 16-21: 打磨
  - 动画/音效/UI 适配
  - 接入你的 settings 页面管理
  - PWA 缓存
```

## 附录 A：Steam 游戏中值得借鉴的设计模式

这次搜索扩展到 Steam 平台，找到了大量将知识/学习融合进游戏的成功案例。以下是按"设计理念"整理的精华，不罗列游戏名，直接提取**你可以拿走用的核心机制**。

### 模式 1：答错 ≠ 失败，答错 = 剧情分支

**代表游戏：Majotori**（Steam 92% 好评，单人叙事答题游戏）

传统答题游戏：答对得分，答错不得分 → 无聊。
Majotori 的做法：**每道题的答案（无论对错）都会推动剧情分支。** 答对推进 Hero 路线，答错推进 Tragic 路线。玩家不是怕答错，而是好奇"如果我选这个会怎样？"

**→ 你的应用**：答对 = 防御塔升级 / 新概念解锁。答错 = 激活 AI 导师解释 / 进入 Socratic 模式。**答错不是惩罚，是另一条学习路径。**

### 模式 2：答题 = 生存筹码

**代表游戏：Trivia Murder Party**（Jackbox 系列最受欢迎的游戏之一）

游戏形式：一群人答选择题，答错的人被"杀掉"（动画里被锯子锯）——答对才能活到下一轮。中间插 mini-game 调节节奏。

**→ 你的应用**：卡片对战里答对 = 攻击，答错 = 敌人攻击 + 扣血。**"我只有 3 条命"比"我要拿高分"的驱动力强 10 倍。**

### 模式 3：答题速度 = 攻击力

**代表游戏：Trivia Throwdown**

答题对战的格斗游戏。你答对的**速度**决定你的角色出拳的**伤害**。1 秒内答对 = 重拳，5 秒后答对 = 轻推。

**→ 你的应用**：倒计时与伤害挂钩。15s 时限内越快答出，攻击力越高。鼓励"立刻回忆"而不是"翻笔记"。

### 模式 4：答错触发 debuff（反常规设计）

**代表游戏：RPG Quiz**（2025 年 3 月发行，免费，Steam 评测良好）

敌人在你答错时可以对你施加状态效果：
- **眩晕**：下一题选项乱序
- **致盲**：隐藏部分选项
- **混乱**：你选的答案会被反转（选 A 变成选 B）

**这个设计极其聪明——答错不是游戏结束，而是"游戏变得更难更有趣"。**

**→ 你的应用**：答错后 next question 更变态，而不是游戏结束。错误驱动学习。

### 模式 5：学一门语言，而不是背单词（启示录级设计）

**代表游戏：Chants of Sennaar**（Steam 96% 好评，2023 年 TGA 提名）

整个游戏没有一道"考题"。你在一个古文明遗迹中探索，通过**观察上下文**来破译一门未知的文字。比如看到"门"旁边有一个符号，门的另一边画着"出口"，你就推导出这个符号是"门/O"的意思。

**这触及了"在游戏中学习"的终极哲学：最好的学习体验根本不像学习。**

**→ 你的应用**：不要只做"选择题题库"。设计一些需要玩家**从 wiki 知识中推导出答案**的任务。例如：给你两个概念的定义，让你推测它们之间的关系（wiki_links 就是正确答案）。这种"推导"比"四选一"的记忆深 3 倍。

### 模式 6：用错误记忆更多（反直觉）

心理学上的"好错误效应"（Desirable Difficulties）：如果学习过程需要付出努力，甚至犯错，长期记忆反而更强。

**→ 你的应用**：在游戏里**主动创造"难忘的错误"**。比如答错塔防题，敌人会嘲讽你一句相关的知识梗。玩家记住的不是"我答错了"，而是"那个冷笑话说 XX 概念其实不是 XXX"。

### 小结：从 Steam 提取的 6 条设计法则

```
1. 答错不是失败，是分支剧情        ← Majotori
2. 答题就是生存，死了才有紧迫感     ← Trivia Murder Party
3. 速度 = 力量，鼓励快速回忆        ← Trivia Throwdown
4. 答错 = 敌人变强/debuff         ← RPG Quiz
5. 最好的学习根本不像是学习          ← Chants of Sennaar
6. 设计难忘的错误，而不是无痛的失败   ← 认知科学
```

这些设计模式完全可以组合进你的 Card Battle 方案里。下一版迭代规格（Phase A 实现文档）时会把这些机制写进具体的 Sprint 任务。
